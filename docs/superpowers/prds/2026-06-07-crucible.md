# PRD: Crucible — self-improving text-to-SQL agent builder

**Status:** ready-for-agent (published to GitHub Issues on Aankirz/Crucible)
**Date:** 2026-06-07
**Related:** design spec `docs/superpowers/specs/2026-06-07-crucible-design.md` · implementation plan `docs/superpowers/plans/2026-06-07-crucible.md`
**Context:** Google Cloud Rapid Agent Hackathon — Arize track.

---

## Problem Statement

A developer with a relational database wants a high-quality natural-language-to-SQL agent for *their* schema. Today that means hand-crafting a prompt, guessing which few-shot examples help, and eyeballing whether the results are right — a slow, manual, opinion-driven loop with no objective measure of "is this actually better than what I had an hour ago?" There is no trustworthy, automated way to know whether a change to the agent improved accuracy or just moved the failures around, and no record of *why* each change helped. The developer ends up shipping a text-to-SQL agent they can't defend with a number.

## Solution

Crucible turns that manual grind into an autonomous, evidence-backed loop. The developer points Crucible at a database; Crucible builds a candidate text-to-SQL agent, scores it objectively against a held-out evaluation set using **execution match** (does the generated SQL return the same rows as the known-correct SQL?), then reads its *own* run traces through the Arize Phoenix MCP server to diagnose the dominant failure pattern, proposes one targeted improvement, and re-scores. It repeats — each version a visible rung on a leaderboard — until it clears a quality bar or stops improving. The developer keeps oversight via approval gates and a final promotion step. The output is a tuned, deployable agent for that specific database, accompanied by a defensible accuracy number on data the agent never saw during tuning.

The core promise: **one database in → one tuned, measured text-to-SQL agent out**, built by an agent that improves itself from its own observability data.

## User Stories

1. As a developer, I want to point Crucible at a database, so that I get a text-to-SQL agent specialized for that schema without hand-tuning prompts.
2. As a developer, I want Crucible to evaluate candidate agents against known-correct SQL, so that "better" is an objective number, not an opinion.
3. As a developer, I want scoring to use execution match (same result rows) rather than exact SQL-string match, so that a correct-but-differently-written query still gets credit.
4. As a developer, I want the evaluation set split into a set the agent learns from and a held-out set it is scored on, so that the reported accuracy reflects generalization, not memorization of the examples.
5. As a developer, I want the evaluation set weighted toward harder queries, so that the headline score has real room to improve and isn't saturated by trivial lookups.
6. As a developer, I want each optimization round to make exactly one targeted change, so that I can attribute any accuracy gain to a specific decision.
7. As a developer, I want Crucible to diagnose the dominant failure category each round (e.g. join errors, aggregation errors, schema/column hallucination), so that improvements address the biggest problem first.
8. As a developer, I want Crucible to read its own run traces to form its hypotheses, so that improvement is driven by real observability data rather than guesswork.
9. As a developer, I want a candidate accepted only when it improves on the learning set, so that the agent never regresses into a worse version.
10. As a developer, I want Crucible to always keep the best-scoring version, so that an unlucky late round can't lose me my best agent.
11. As a developer, I want the loop to stop when it hits the quality bar, the iteration cap, the cost budget, or two rounds without improvement, so that it never runs away or burns unbounded budget.
12. As a developer, I want candidate scoring to be deterministic (fixed temperature), so that the same version always produces the same score and the leaderboard is trustworthy.
13. As a developer, I want a read-only, time-bounded SQL sandbox, so that a generated query can never modify my database or hang the run.
14. As an operator, I want to watch the optimization happen live on a leaderboard, so that I can see the held-out score climb in real time.
15. As an operator, I want to see the current hypothesis and its rationale each round, so that I understand *why* the agent is changing itself.
16. As an operator, I want an autopilot mode and a step-by-step approval mode, so that I can let it run for speed or intervene for control.
17. As an operator, I want to approve or reject a proposed change, so that I stay in control of what the agent does.
18. As an operator, I want to explicitly approve the final promotion, so that nothing ships without my sign-off.
19. As a developer, I want the promoted agent persisted to the Phoenix prompt registry, so that the winning version is versioned and retrievable.
20. As a developer, I want the promoted agent exportable as a runnable agent, so that I can deploy what Crucible produced.
21. As a developer, I want to run Crucible on a different database and get a different tuned agent, so that the approach demonstrably generalizes across schemas.
22. As a developer, I want to see results from the same loop on multiple databases, so that I can trust it isn't hand-tuned for one schema.
23. As a developer, I want the agent's introspection to flow through a standard MCP interface, so that the observability integration is real and portable, not a bespoke back-channel.
24. As a developer, I want every candidate run traced, so that I have a complete, inspectable record of the build.
25. As a developer, I want each scored version logged as an experiment, so that the leaderboard is backed by durable, comparable records.
26. As a developer, I want failures that error out (bad SQL) to score zero and capture the error, so that the error text becomes a high-signal input to the next hypothesis.
27. As a presenter, I want a one-database live demo plus pre-computed results for two more databases, so that I can show both the live climb and proof of generality within a short slot.
28. As a presenter, I want to show the real observability dashboard alongside the custom view, so that judges can verify the integration is genuine.
29. As a developer, I want the whole core loop to run and be tested without any credentials or network, so that I can build and verify the logic before wiring external services.
30. As a developer, I want the system deployable to a managed runtime, so that it satisfies a code-owned-runtime requirement and can be shared via a URL.
31. As a developer, I want a baseline first version that has the schema but no examples, so that the loop's improvements (added examples, refined instructions) are the visible driver of the climb.

## Implementation Decisions

- **Scope is deliberately bounded to benchmark databases (Spider/BIRD).** These ship human-authored gold SQL, so ground truth is always trustworthy and the "no ground truth on an arbitrary DB" problem is avoided entirely. Crucible never authors the gold it grades against.
- **Mechanism is self-improving, output is a deployable artifact.** Crucible improves a text-to-SQL capability by reading its own run traces; the promoted artifact (tuned prompt + few-shots + tool/context config) is a deployable agent for that database. One database → one tuned agent (specialization is per-schema because the highest-ROI levers are schema-specific).
- **Optimization strategy is reflexive/hypothesis-driven** (hill-climbing with reasoning), chosen over automated optimizers or evolutionary search for transparency, low cost, and a legible narrative.
- **Modules built (responsibilities, not paths):**
  - *Comparator* — execution-match decision over two result sets. Pure.
  - *SQL sandbox* — read-only, statement-timeout-bounded query execution; returns rows or an error string.
  - *Dataset pipeline* — load benchmark gold; classify difficulty; stratified train/test split; difficulty-weighted sampling. Pure.
  - *Candidate runtime* — render a prompt from a candidate spec; extract SQL from model output; run an item via an injected model function.
  - *Eval engine* — score a candidate over a split, producing per-item results and an aggregate.
  - *Mutation engine* — classify each failure into a fixed taxonomy, pick the highest-impact cluster, propose one atomic hypothesis (instruction + few-shots), and apply it immutably.
  - *Observability/MCP layer* — deterministic plumbing (tracing, experiment logging, prompt registry) via the Phoenix SDK; decision-critical introspection via agent-initiated Phoenix MCP tool calls.
  - *Orchestrator* — the loop tying the above together with accept/revert, stopping criteria, best-so-far, and an event stream.
  - *Mission Control* — an event-streamed UI showing leaderboard, current hypothesis, and approval/promotion controls.
- **Key interface decisions (encoded as type shapes from the design, more precise than prose):**

  A candidate agent is fully described by an immutable spec:
  ```
  CandidateSpec { version, system_prompt, few_shots[(question, sql)], enable_schema, enable_validate_tool }
  ```
  All LLM behavior is injected as a single function seam `ModelFn: prompt -> text`, so the loop is testable without network.
  The loop's accept/report rule (encodes the anti-overfitting decision): **mutations are accepted on the train split's score; the held-out test split's score is the reported leaderboard number; the best test-scoring version is always retained and promoted.**
- **Stopping is governed by an explicit config:** target score (default 90%), max iterations (default 6), patience (stop after 2 non-improving rounds → revert to best), plus a cost/token budget.
- **Determinism:** candidate generation runs at temperature 0 so scores are reproducible; the comparator normalizes numeric tolerance, trims strings, is NULL-aware, treats rows as a multiset, and is order-sensitive only when the gold query specifies an order.
- **MCP integration is hybrid and meaningful:** the agent itself calls the Phoenix MCP server to read its own failing runs and the experiment leaderboard (driving each hypothesis) and to promote the winner — while deterministic plumbing stays in code so the demo can't flake. The introspection layer exposes one stable interface (`introspect(experiment_name) -> summary`) so the transport can change without touching the loop.
- **Runtime:** a code-owned agent runtime (Google ADK + Gemini) is required (the visual builder alone cannot be instrumented for tracing). Deployment target is a managed container runtime (Cloud Run); local-first for demo reliability.
- **Generality is demonstrated with three databases:** one optimized live, two pre-computed, shown as a results table.

## Testing Decisions

- **What makes a good test here:** assert *observable behavior of the loop*, never implementation details. Good assertions: the held-out score improves across versions; a candidate that doesn't beat the train score is rejected and the prior spec is retained; the best-scoring version is the one promoted; the loop early-stops after the patience threshold; a malformed query scores zero and records its error. Bad assertions (to avoid): exact prompt string contents, internal call counts, or specific few-shot wording.
- **Seams (confirmed):**
  1. **`ModelFn` (`prompt -> text`)** — the primary, highest seam. Deterministic fake models drive the entire loop end-to-end with no network, including a fake whose answers flip *after* a mutation, to prove the loop genuinely improves.
  2. **`run_loop` callback seams** — `introspect`, `log_experiment`, `on_event` — stubbed to isolate external effects so the loop's behavior is asserted in pure form.
  3. **SQL sandbox** — exercised for real against a temporary SQLite fixture DB (fast, deterministic; higher fidelity than a mock).
  4. **Pure functions** — comparator, difficulty classifier, split/sampling, failure classification, cluster pick, prompt rendering, SQL extraction, hypothesis application — tested directly.
- **Modules tested:** comparator, sandbox, dataset pipeline (classifier + loader + split), candidate runtime, eval engine, mutation engine, orchestrator loop. The observability/MCP layer and UI are verified via an end-to-end smoke run rather than unit tests (network/LLM-bound).
- **Prior art:** none in-repo (greenfield). The test style follows standard pytest fixtures with temp SQLite databases and injected fakes; the comparator and loop tests are the reference patterns subsequent tests should imitate.
- **Coverage intent:** the deterministic core (Phases 1–3, 5–6 of the plan) is the TDD spine and should be near-fully covered before any external service is wired.

## Out of Scope

- Arbitrary user databases with no gold SQL (and the associated bootstrapped/consensus-validated ground-truth machinery). Bounded to Spider/BIRD.
- Mutating model choice or temperature, or having the agent author entirely new tools/code. Levers are limited to prompt instructions, few-shot exemplars, and enabling existing tools/context. (Tool/param mutation and an evolutionary "turbo" mode are explicit stretch goals, not in this PRD.)
- Authentication, multi-tenancy, and any persistence beyond Phoenix (datasets/experiments/prompts/traces).
- Production hardening of the deployed service beyond a working demo URL.
- Non-SQLite execution backends (Spider/BIRD ship SQLite, which is sufficient).

## Further Notes

- This PRD aligns to the Arize track rubric: technical implementation, *meaningful* tracing + MCP, *quality of the self-improvement loop*, and impact. The deliberate design choice that wins the bucket is making the decision-critical introspection genuinely agent-initiated through MCP, with the self-improvement loop as the entire product rather than tracing bolted onto a chatbot.
- The riskiest integration (ADK ↔ Phoenix MCP wiring) is de-risked first via a throwaway spike before anything depends on it; a documented Gemini-CLI fallback transport exists if the toolset path proves unworkable. The loop depends only on the stable `introspect` interface, so this risk is contained.
- Honesty safeguards baked in: held-out test split (anti-overfitting), human-authored gold (no self-grading), execution match (credits correct-but-different SQL), and temperature 0 (reproducible scores). These are the points a sharp judge will probe, and each has a one-sentence answer.
